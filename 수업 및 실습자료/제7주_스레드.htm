2.              (Worker Thread)

 

이번에는 힘들고 어려운 일을 별도의 스레드가 뒷 마당에서 눈에 띄지 않게 하도록 하고, UI 스레드는 손님 시중 드는 일에 전념하도록 해 봅니다. 손님 마음에 들지 않으면 손님이 아마도 앱을 지워버릴 것입니다.

새 애플리케이션을 만드시오. 이름은 "WorkerThread"로 지으시오. 화면 구성은 위 1번과 비슷하지만 버튼 하나를 아래 그림과 같이 추가합니다.

 

MainActivity 자바 코드도 위 1번과 크게 다르지 않지만 힘든 일을 별도의 스레드가 하게 하는 부분만 다릅니다.

"힘든 작업" 버튼을 클릭하면 별도의 스레드(worker thread)가 만들어지고 그 스레드가 힘든 작업을 10초 동안 수행하고 상태변수 hiddenCount 값을 1 증가 시킵니다. hiddenCountView에 값을 출력하고 싶지만 그렇게 할 수는 없습니다. 오직 UI 스레드만이 UI 화면의 뷰들에 접근할 수 있기 때문입니다. 그래서 대신 액티비티의 상태변수인 hiddenCount 값을 증가시키도록 하는 것입니다.

Worker thread가 hiddenCount 값을 증가시켜도 이 값이 화면에 보이지는 않습니다. 사용자가 "변수 보기" 버튼을 클릭하면 hiddenCount 값을 hiddenCountView에 출력하도록 합니다.

 

앱을 실행하고 "증가" 버튼을 몇 번 클릭하여 작동을 확인합니다.

"힘든 작업" 버튼을 한 번 클릭합니다. 그러면 worker thread가 열라 힘들고 어려운 일을 10초 동안 수행한 후 hiddenCount를 1 증가시킵니다. worker thread가 일을 시작할 때 UI 스레드가 화면에 토스트 메세지를 보여줍니다.

10초가 지나기 전에 "증가" 버튼을 몇 번 클릭해 봅니다. Worker thread가 힘든 일을 하는 동안에도 사용자 시중을 드는 UI 앱은 빠릿빠릿하게 일을 잘 합니다.

10초가 지났는지 궁금하니까 "변수 보기" 버튼을 반복해서 클릭해 봅니다. 때가 되면 (10초가 지나면) hiddenCountView에 만큼 증가한 값이 출력됩니다.  Worker thread가 힘든 일을 하는 동안에도 사용자 시중을 드는 UI 앱은 빠릿빠릿하게 일을 잘 합니다.

 

다시 "힘든 작업" 버튼을 여러 번 한 번 클릭합니다. 토스트 화면이 사라지기 전에 연달아 클릭해도 됩니다. 토스트 화면은 시스템이 출력할 내용을 자기 버퍼에 저장해 주고 차례로 출력해 주는 것으로서 앱 스레드 흐름과는 별개로 돌아갑니다. 이어서 "증가" 버튼을 눌러 봅니다. 이렇게 해도 반응성은 여전히 좋습니다. 이 경우에는 여러 개의 worker thread들이 병렬로 힘든 일을 하고 있는 상태입니다.

여러 개의 worker thread들이 일을 하는 동안 "변수 보기" 버튼을 계속 클릭하다 보면 worker thread들이 일이 끝나고 증가한 hiddenCount 값이 화면에 나타나게 됩니다.

 

여러 개의 worker thread들이 하는 일들이 요구하는 CPU, 메모리 요구량의 총합이 기기의 CPU, 메모리 양에 비해 적절한 수준이라면 앱은 빠른 반응성을 유지합니다. UI 스레드가 다른 스레드에비해 CPU, 메모리 등 자원 배분에 있어 우선권을 갖기 때문입니다.

 

(참고)

스레드를 시작시키는 방법은 두 가지가 있습니다.

①              Ruunable 객체를 생성하고 이 객체를 인자로 주면서 Thread를 생성한 후 start시킨다.

②              Thread의 서브클래스를 작성하되 그 내부에서 run 메소드를 재정의한다. 그리고 이 서브클래스 객체를 생성하고 start시킨다.

우리는 이 중 첫 번째 방법을 사용했습니다. 자세한 사항은 인터넷 검색을 하거나 자바프로그래밍 강의자료를 찾아보세요.

 

3.              (Sychronization)

 

동기화에 대한 기본 지식을 복습하는 문제입니다. 여러 개의 스레드가 병렬로 돌면서 각각이 스레드가 어떤 객체에 들어 있는 상태변수를 변경하는 경우 그 상태변수의 데이터가 잘 못 될 수 있습니다. 이런 상황이 발생하는 조건을 레이스컨디션(race condition, 경쟁조건)이라고 부릅니다. 이 문제가 발생하지 않게 하려면 스레드들 가나에 동기화 (synchronization) 작업을 해 주어야 합니다. 동기화는 기법을 실습하기 전에 우선 스레드들 간에 경재조건이 만들어져 데이터가 훼손되는 현상을 만들어 관찰해 봅니다.

경쟁조건이 만들어지려면 스레드 스위칭(CPU가 한 스레드를 실행하다가 다른 스레드로 바꿔 실행하는 일)이 일어나는 순간 스레드가 상태변수 값을 갱신하는 일을 하고 있었어야 합니다. 이 확률이 매우 낮기 때문에 많은 반복을 해야 이 현상을 만들 수 있습니다.

새 애플리케이션을 만드시오. 이름은 "Synchronization"으로 지으시오. 화면 구성은 위 2번과 같습니다. 자바 코드도 거의 비슷하지만 스레드가 자주, 그리고 매우 여러 번, 불규칙한 시간 간격으로hiddenCount 값을 증가시키도록 합니다.

 

앱을 실행하고 "힘든 작업" 버튼을 연달아 다섯 번 클릭한다. 그러면 스레드 다섯 개가 동시에 돌면서 각각 hiddenCount를 100번씩 증가시킨다. "변수 보기"를 클릭하면 500이 보여야 하지만 실제로는 500 근처의 다른 수가 나타난다. 만약, 500이 나타났다면 다시 같은 실험을 반복해 본다.

이런 문제가 발생하지 않게 하려면 한 스레드가 변수를 변경하는 중에는 스레드 스위칭이 일어나지 않게 해야 한다. 동기화를 해 주어야 한다. 아래 그림과 같이 자바 코드를 수정한다.

이렇게 수정한 후 앱을 실행하여 위에서 한 실험을 해 본다. 여러 번 반복해도 항상 정확히 100의 배수가 나타나는 것을 볼 수 있을 것이다.

4.              (SimpleAsychTask)

 

1번부터 3번까지 표준 자바에서 제공하는 multithreading과 synchronization을 복습했습니다. 이 문제에서는 안드로이드에서 제공하는 클래스인 AsychTask를 사용해 봅니다. AsychTask는 시간 걸리는 일을 worker thread로 실행시키면서 UI thread에서 중간 중간 진행상황을 보고받아 진행상황을 화면에 보여주고, worker thread의 작업이 끝나면 그 결과를 보고 받아 화면에 보여주는 일을 하고자 할 때 편리하게 사용할 수 있는 높은 레벨의 클래스입니다. 사용하기 편리하지만 정해진 목적에만 사용할 수 있는 제약이 있습니다. 몇 초에서 몇 십 초 정도 걸리는 작업에 적합하며 AsynchTask 객체를 생성하고 나면 단 한 번만 execute시킬 수 있습니다. 몇 초에서 몇 십 초 정도 걸리는 작업에 적합하다고 하는 이유는 AsynchTask가 시스템 공동으로 사용하는 thread pool에 미리 만들어져 있는 thread를 이용하기 때문입니다. thread pool에 미리 만들어져 있는 thread 수가 정해져 있는데 한 놈이 thread를 차지하고 너무 오래 사용하면 다른 놈들이 뭐라 할 것이기 때문입니다.

 

새 프로젝트를 만들고 애플리케이션 이름을 SimpleAsynchTask로 지으시오.

세 개의 필드가 필요합니다.

(1) ListView를 사용할 때는 Adapter가 필요합니다. 기본적으로 많이 사용하는 Adapter는 ArrayAdapter입니다. ArrayAdapter는 타입 파라미터를 갖는 generic class입니다. 우리는 과목이름 String들을 아이템으로 넣을 것이므로 ArrayAdapter<String>을 사용합니다.

그러니까 아래와 같은 필드가 우선 필요합니다. 처음에는 null로 초기화합니다.

ArrayAdapter<String> adapter = null;

(2) 우리가 사용할 ArrayAdapter 생성자 형식은 아래와 같습니다. 세 번째 파라미터가 List입니다.

그러니까 과목명을 담을 List<String>을 만들고 그것을 인자로 주면서 ArrayAdapter<String> 객체를 생성해야 합니다. 그래서 아래와 같은 필드가 필요합니다. ArrayList<String> 객체로 초기화합니다. 초기화를 했다고 해도 아직 과목 이름 데이터가 이 List에 들어 있는 것은 아닙니다.

List<String> model = new ArrayList<String>();

이 List에 String들을 넣어 주면 그들이 ListView에 보여지게 됩니다. 나중에 worker thread에서 하나씩 하나씩 과목 이름 String을 보내오면 그 때마다 이 리스트에 추가할 것이고 그러면 차례로 화면에 나타날 것입니다. 리스트에 추가할 때는 직접 리스트에 add 명령을 내리는 대신 adapter에게 add 명령을 내립니다.

(3) 우리는 worker thread가 과목이름을 (멀리서, 그러나 지금 우리의 간단한 예에서는 배열에서) 가져오게 하고 그 내용을 UI thread가 받아 화면에 나타나게 할 계획입니다. 이를 위해 AsychTask를 사용할 것입니다. 우리가 만들 AsychTask의 서브클래스 이름을 AddStringTask로 지읍시다. 그러면 AddStringTask 객체를 가리킬 필드가 필요합니다. 일단 null로 초기화합니다.

AsynchTask의 세 개의 형식 파라미터는 Params, Progess, Result입니다. 일을 시킬 때 특별히 넘겨 줄 정보가 없으므로 Params는 Void로 설정했습니다. 중간 중간 성과 보고를 할 때는 리스트에 들어 있는 과목 이름을 보고해야 합니다. 그러니까 Progess는 String 타입입니다. 일을 마치고도 특별히 보고할 게 없습니다. 그러니까 Result도 Void로 정해 주었습니다.

AddStringTask에서 우선 할 일은 worker thread에서 실행될 doInBackground 메소드를 재정의하는 것입니다. 태스크가 시작될 때 아무런 값도 받아 오지 않으므로 파라미터는 Void입니다. 단, varargs 형태로 적어 주어야 하므로 아래와 같이 적어야 합니다.

Thread.sleep 메소드는 스레드를 잠들게 하는데 잠자는 동안 누군가가 이 슬레드에게 인터럽트를 걸면 (thread.interrupt 메소드를 호출함으로써) InterruptedException이 발생하는데 이게 확인예외(checked exception)입니다. 그래서 예외처리를 해 주지 않으면 컴파일이 되지 않습니다. 위 코드에서는 인터럽트가 걸리면 일을 중간에 끝내도록 해 주었습니다.
