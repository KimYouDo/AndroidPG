2.     (FileIO) – 외부 저장소 (개별스토리지)

 

1번 문제를 복사하여 FileIO2로 이름 짓는다.

1번 문제의 onSaveData와 onLoadData를 수정한다. 1번 문제에서는 getFilesDir()을 통해 내부 저장소의 경로를 가져왔는데, 2번 문제에서는 getExternalFilesDir()를 이용해 외부 저장소에 접근한다. getExternalFilesDir()는 외부 저장소의 private 영역의 경로를 가져온다. 외부 저장소의 private 영역에 파일을 저장하면, 다른 앱이나 사용자가 이 파일에 접근할 수 없으며, 앱이 삭제될 때 그 파일도 같이 삭제된다.

외부 저장소에 접근할 때는 그 저장소가 현재 기기에 연결되어 있는지 확인하는 작업이 필요하다. 외부저장소가 기기 내장 플래시 메모리가 아닌 확장형 SD카드일 수도 있기 때문이다.

또, 앱에 외부 저장소에 접근하려면 manifest에서 그 권한을 부여해 줘야 한다.

 

먼저 Manifest 파일에서 권한을 부여해 준다.

 

Manifest 파일을 딱히 수정하지 않았으면 <application>안에 <activity>정보만 있을 것이다.

다음과 같이 외부 저장소에 대한 읽기와 쓰기 권한을 허용해 준다.

 

이제 외부 저장소를 사용할 수 있게 되었다. 다음으로는 코드에서 외부 저장소에 접근하기 전에 외부 저장소가 기기에 연결되어 있는지 확인하게 한다. 이를 위해 Enviroment.getExternalStorageState() 메소드를 사용한다.

getExternalStorageState()메소드가 반환하는 외부 저장소의 상태에는 다음과 같은 것들이 있는데, 장치가 연결되어 있으면 MEDIA_MOUNTED가 반환된다. MEDIA_MOUNTED 상태에 있는 외부 저장소는 읽기와 쓰기가 모두 가능하다. MEDIA_MOUNTED_READ_ONLY 상태에 있는 외부 저장소는 읽기만 가능하다.

 

외부 저장소에 파일을 쓸 수 있는지 검사하는데 사용하는 메소드를 아래 그림과 같이 작성하고, 파일을 쓸 때는 이 메소드를 이용해 쓰기가 가능한지 여부를 검사한다.

다음은 외부 저장소의 파일을 읽을 수 있는지 검사하는데 사용하는 메소드이다.

 

외부 저장소에 파일을 쓰기 위해서는 getFilesDir 메소드 대신 getExternalFilesDir 메소드를 사용하는데 getExternalFilesDir 메소드를 호출할 때는 인자를 하나 주어야 한다.

 

위 그림에서 보듯이 String형식의 type값을 가지는데 이는 외부 저장소에 저장될 폴더를 의미한다. 만약 null을 주면 외부 저장소의 root에 저장된다.

우리는 null을 사용하여 외부 저장소 root경로에 저장한다.

####중요####

만약 자신의 에뮬레이터 혹은 기기의 버전이 API 23 미만이면 지금까지 작성한 앱이 정상적으로 작동한다. 그러나 API 23 (Marshmallow) 이후라면 제대로 작동하지 않는다.

API 23부터는 public 외부 저장소 접근 권한은 “위험 권한”으로 분류되어 강화된 보안 정책이 적용되기 때문이다.

앱의 manifest에 외부 저장소에 대한 접근 권한을 갖도록 적어 주면 앱이 설치될 때 사용자에게 외부 저장소 접근 권한을 허용할지 물어 보게 된다. 이에 대해 사용자가 권한을 허용할 때만 앱이 기기에 설치된다. 그런데 많은 사용자가 앱 설치 때 나타나는 여러 가지 권한 요청에 대해 신중히 생각하지 않고 무조건 허용하는 경우가 많아 보안 문제가 발생하는 수가 있다. 이에 대한 대처 방안으로 새로 도입된 것이 바로 앱 실행 중 추가 권한 획득이다. Manifest를 통해 앱 설치 시 권한을 얻었다고 하더라도 그 권한이 “위험 권한”인 경우에는, 실제로 앱이 실행되어 그 권한을 처음으로 사용하려고 할 때, 다시 한 번 사용자의 확인 허락을 받도록 하는 것이다.

 

또, 앱은 사용자로부터 어떤 “위험 권한”에 대해 한 번 허락을 받았다고 하더라도 그 권한을 사용할 때마다 여전히 그 권한이 유효한지 체크해야 한다. 사용자는 언제든지 그 권한을 회수할 수 있기 때문이다.

 

실행 중 사용자로부터 권한을 얻는 방법은 조금 복잡하므로 실제로 필요할 때 공부해서 사용하기로 하고 실습을 하지는 않기로 한다.

 

실행 중 위험 권한에 대해 권한을 요청하고 사용자 응답을 받아 처리하는 방법에 대해 잘 알고 싶다면 아래 글을 참고한다.

안드로이드에서 SQLiteDatabase를 사용할 때는 SQLiteOpenHelper를 함께 사용한다. 강의자료 중 SQLiteOpenHelper에 관한 내용을 다시 한 번 정독하고 이 과제를 시작하도록 한다.

 

새 프로젝트를 만든다. 애플리케이션 이름은 Constants1으로 짓는다.

다음으로 프래그먼트 클래스를 작성한다. 클래스 이름을 ConstantsFragment로 짓고 ListFragment의 서브클래스로 만든다.

 

이 클래스에서 할 일은 데이터베이스를 읽어 화면에 보여 주는 일이다.

 

ListFragment는 내부에 ListView를 가지고 있다. FragmentList는 스스로 ListView를 만들기 때문에 onCreateView 메소드를 재정의하지 않아도 된다. (onCreateView는 레이아웃을 부풀려 View를 만드는 데 주로 사용한다.)

 

일반적으로 ListView에 아이템들이 나타나게 하려면 adapter를 만들어 ListView에 등록해야 한다.

그러나 ListFragment의 경우에는 ListFragment에게 adapter를 등록하면 된다. 그렇게 하면 내부적으로 이 adapter가 ListView에 연결된다.

 

데이터베이스를 읽으면 Cursor가 반환된다. Cursor에는 데이터베이스에서 읽은 데이터가 행과 열 형태로 들어 있다.

Array에 들어 있는 데이터를 ListView에 넣어 주는 일을 하는 클래스가 ArrayAdapter였다. Cursor에 들어 있는 데이터를 ListView에 넣어 주는 일을 하는 클래스는 CursorAdapter이며, 그 서브클래스로서 쉽게 사용할 수 있는 것이 SimpleCursorAdapter이다. ArrayAdapter가 내부에 array를 갖고 있듯이 CursorAdapter는 내부에 Cursor를 갖는다. Array나 Cursor는 초기에 null일 수 있다. 나중에 array나 cursor를 넣어 주면 그 내용이 ListView에 나타난다.

 

프래그먼트 뷰가 만들어진 후에야 비로소 ListView에 adapter를 등록할 수 있다. 그러니까 뷰가 만들어지 후에 호출되는 메소드, 즉 onViewCreated 메소드 혹은 onActivityCreated 메소드에서, 프래그먼트에 adpater를 등록하고 데이터베이스를 읽어 adapter에 넣어 주면 된다. 처음 데이터베이스를 읽을 때 SQLiteOpenHelper가 작동하여 데이터베이스가 만들어지고 초기 데이터가 채워 진다.

데이터베이스를 만들거나 구조를 바꾸는 작업은 비교적 시간이 많이 걸리는 일이다. 데이터베이스에 데이터를 쓰거나 읽는 일도 데이터의 크기가 크면 시간이 많이 걸릴 수 있다. 따라서 데이터베이스 작업은 원칙적으로 worker thread에서 이루어져야 한다. 이 문제에서는 위 5번 문제에서 구현한 앱을 개선하여 데이터베이스 접근 작업이 worker thread에서 이루어지도록 한다. 지난 주에 배운 AsynchTask를 이용한다.

 

Constants3라는 이름으로 새 프로젝트를 만들고 Constants2의 파일들을 복사해 온다.

새 프로젝트를 실행하고 화면에 나타난 데이터를 보자. 5번 문제에서 대화창을 통해 추가한 데이터는 보이지 않는다. 새 앱에서 기존 앱과 같은 이름의 데이터베이스를 사용하지만 전혀 다른 디렉토리에 있는 별개의 데이터베이스를 새로 만들어 사용하는 것이기 때문이다.

 

ConstantsFragment.java 파일을 열고 코드를 추가한다.

우선 AsynchTask 필드를 추가한다.

다음으로는 기기 회전 때 프래그먼트가 소멸하지 않도록 설정한다. 이 작업은 onCreate에서 해 준다.

 

onViewCreated 메소드에서는 adapter를 만들어 ListFragment에 설정한 후, SQLiteOpenHelper 인스턴스를 구성하고 데이터베이스로부터 데이터를 읽어 와 adapter에 넣어 주는 일을 했다. 이 일 중 데이터베이스에서 데이터를 읽어 오는 일을 worker thread가 하도록 변경한다.

LoadCursorTask는 AsynchTask의 서브클래스로서 앞으로 작성해야 한다.

 

데이터 항목 추가를 위한 대화창에서 “OK” 버튼을 클릭하면 onClick 메소드가 실행된다. 이 메소드에서 새 데이터를 데이터베이스에 추가하고 데이터베이스를 새로 읽어 화면을 갱신한다. 이 작업도 worker thread에서 실행되도록 변경한다.

InsertTask도 AsynchTask의 서브클래스로서 앞으로 작성해야 한다. InsetTask.execute를 호출할 때는 어떤 데이터를 데이터베이스에 삽입할 것인지를 알려 주어야 하므로 ContentValues를 인자로 넘겨준다. InsertTask는 데이터 항목을 데이터베이스에 삽입한 후, 갱신된 데이터베이스 테이블을 읽어 그 결과를 반환하고 화면에 보이게 한다.

 

AsynchTask의 서브클래스를 작성할 때는 doInBackground, onProgressUpdate, onPostExecute 등 세 개의 메소드를 주로 재정의한다. 우리의 경우에는 데이터베이스 작업이 진행되는 중간에 중간보고를 받을 필요가 없으므로 onProgressUpdate 메소드는 재정의하지 않는다.

LoadCursorTask와 InsertTask 등 두 클래스에 공통으로 적용되는 사항이 있고 그렇지 않은 사항이 있다. 데이터 항목을 삽입하는 일은 InsertTask에서만 필요하다. 반면, constants 테이블 전체를 읽어 adapter에 넣어 주는 일은 두 클래스 공통이다. 우리는 두 클래스 공통 작업을 구현한 AnsychTask의 서브클래스를 먼저 구현하고, LoadCursorTask와 InsertTask 등 두 클래스는 다시 그 서브클래스로서 차이가 있는 기능만을 각각 추가로 구현하는 방법을 사용한다.

 

우선 아래 그림의 BaseTask 클래스를 먼저 작성한다. AsynchTask<Params, Progress, Result>의 서브클래스로 작성한다.

 

LoadCursorTask와 InsertTask 클래스는 이 클래스의 서브클래스로 작성할 것이다. LoadCursorTask와 InsertTask 두 클래스 모두 중간보고는 없으므로 Progress 타입 파라미터는 Void로 고정한다. 또, LoadCursorTask와 InsertTask 두 클래스 모두 백그라운드 작업 실행 결과로서 Cursor를 반환할 것이므로 Result 타입 파라미터는 Cursor로 고정한다. 그러나, LoadCursorTask에 일을 시킬 때는 아무런 인자도 주지 않는 반면, InsertTask에 일을 시킬 때는 ContentValues를 인자로 주어야 하므로, Params 타입 파라미터는 고정하지 않고 타입 파라미터를 유지한다. 그러니까 BaseTask는 타입 파라미터 하나를 갖는 제네릭 클래스이다. 또, 이 클래스 객체는 만들지 않을 것이므로 abstract 클래스로 선언했다.

 

 

먼저 onPostExectue 메소드를 재정의한다. 이 메소드가 호출될 때는 데이터베이스 읽기의 결과인 Cursor가 파라미터로 넘어 온다. 이 Cursor에 들어 있는 데이터가 화면에 나타나게 해 준다. 이 작업은 LoadCursorTask, InsertTask 공통 사항이므로 이 곳에 적어 준다.

 

잠시 AsynchTask를 놔 두고 onDestroy 메소드를 보자. 아래와 같이 task가 null이 아닐 때는 task를 cancel해 주는 코드를 추가한다. 사용자가 앱을 종료하여 프래그먼트가 소멸한 후에도 worker thread가 하던 일을 계속하도록 놔 둘 필요가 없기 때문이다.

 

위에서 작성한 코드들을 보면 LoadCursorTask나 InsertTask를 만들어 일을 시킬 (execute) 때는 task가 LoadCursorTask나 InsertTask를 가리키도록 했고, 일이 끝나면 onPostExecute에서 task에 null을 넣어 주었음에 주목한다. Task가 진행 중인지 여부를 스스로 모니터링하고 있는 것이다.

 

다시 BaseTask로 돌아와 아래 메소드를 추가한다. 데이터베이스의 테이블을 읽는 일은 LoadCursorTask와 InsertTask 공통으로 필요한 일이므로 이 일을 수행하는 메소드를 이 클래스에 만들어 두고, 두 개의 서브클래스들이 이 메소드를 상속하여 사용하게 한다. 이렇게 하지 않으면 아래 코드가 두 서브클래스들에 중복하여 적혀지게 된다.

 

기존에 있던 ConstantsFragment의 private doQuery 메소드는 이제 필요가 없으니 삭제한다. 5번 프로젝트에서는 UI 스레드에서 이 메소드를 직접 호출함으로써 foreground에서 데이터베이스를 읽는 일을 수행했었는데, 이제 우리는 AsynchTask를 이용해 이 작업이 background에서 실행되게 할 것이다. doInsert 메소드도 마찬가지이다. 삭제한다.

 

이제 LoadCursorTask 클래스를 작성한다. 이 태스크에 일을 시키면 (execute), 아래 그림의 doInBackground 메소드가 worker thread에서 실행된다. 즉, doQuery 메소드가 background에서 실행된다. doQuery 메소드는 Cursor를 반환하는데 doInBackgrond 메소드는 이 Cursor를 받아 그대로 다시 반환한다. 그러면 UI 스레드에서 onPostExecute 메소드가 호출되는데 이 메소드로 Cursor가 넘어 간다.

 

다음으로는 InsetTask를 작성한다. 이 태스크의 doInBackground 메소드는 주어진 데이터 항목을 데이터베이스에 삽입한 후 데이터베이스 테이블을 읽어 Cursor를 반환한다. 그러면 UI 스레드에서 onPostExecute 메소드가 호출되는데 이 메소드로 Cursor가 넘어 간다.

 

앱이 제대로 작동하는지 시험하고 잘 되면 delete 기능도 구현한다.
